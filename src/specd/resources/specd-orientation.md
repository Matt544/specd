# `specd` project orientation

This project uses `specd` to keep specs, tests, and implementation in sync and driven by specs.

- Specs are generated from templates in {{ templates_dir.strip('/' ~ '/') }} (via `*.specd.md` files).
- Canonical specs are under {{ specs_dir.rstrip('/') ~ '/' }} (in `*.md` files).
- Tests that `specd` is aware of are found under {{ tests_dir.strip('/' ~ '/') }}.

The typical process is:
- The human programmer edits specs by editing a `*.specd.md` file. Those permit jinja2 syntax.
- The human programmer runs `specd render` to convert templates into an actual spec file meant to be read and acted on. They are autogenerated into {{ specs_dir.rstrip('/') ~ '/' }}. (For convenience, use `specd render --watch`.)
- Agents only read spec files under {{ specs_dir.rstrip('/') ~ '/' }}. That is the authoritative source for agents, from the standpoint of writing tests and implementations.

Do not edit spec files under {{ specs_dir.rstrip('/') ~ '/' }}, as changes will be overwritten on the next `render`.

Agents should leave the files under {{ templates_dir.strip('/' ~ '/') }} alone, unless specifically directed to make changes.

`specd` assumes compliance with the following policies:
- {{ resources_dir.strip('/' ~ '/') }}spec-writing-policy.md (how to write spec items)
- {{ resources_dir.strip('/' ~ '/') }}test-writing-policy.md (reference format and coverage rules)
- {{ resources_dir.strip('/' ~ '/') }}spec-implementation-policy.md (workflow for implementing specs; unenforced)

Use `specd validate` to check that every test references a spec item and every spec item has a test. (Ensure the venv is activated and run it from within `frontend/` or one of its child dirs.)

`specd` configurations can be changed using a `pyproject.toml` file.

Agents can ignore most aspects of the `specd` CLI, which relate to project setup and template rendering.
